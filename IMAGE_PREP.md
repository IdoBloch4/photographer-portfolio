# Image Preparation Guide

This document describes the exact folder/file/data structure the website
expects for photos. The goal is that preparing a new batch of Sarit's photos
— selecting, sorting into categories, naming — can happen independently of
writing code, and that a future import script can be written against a
predictable, unambiguous spec.

If you're an AI assistant asked to "import more photos" or "write a script
to add these," this document is the spec to write against. See also
`scripts/import-photos.mjs`, which already does exactly this for one batch
and can be extended or used as a template.

## The 5 categories — closed set, don't improvise a 6th

The site organizes all work by **photo genre**, not location or trip. There
are exactly 5 categories. Each is one JSON file in `content/series/` and one
folder in `public/images/series/`, both keyed by the same slug:

| Slug | Title on site | What belongs here |
|---|---|---|
| `landscape` | Landscape | Mountains, valleys, coastlines, wide nature scenes |
| `street` | Street | Urban/documentary, candid city scenes, architecture |
| `macro` | Macro | Close-up / life-size nature detail (flowers, insects, texture) |
| `astro` | Astro | Night sky, long-exposure, Milky Way, star trails, aurora |
| `abstract` | Abstract | Non-representational — color, form, light |

If a photo doesn't clearly fit one of these five, **stop and ask** rather
than inventing a new category. Adding a real 6th category means a new JSON
file, a new image folder, and renumbering `order` in the others — it's a
deliberate site-structure change, not a content add.

## Where files live

```
public/images/series/{category}/{NN}.jpg     ← final, web-ready images
content/series/{category}.json                ← metadata for each image
```

- `{category}` is one of the 5 slugs above (exact spelling, lowercase).
- `{NN}` is a sequential, zero-padded two-digit number starting at `01` —
  `01.jpg`, `02.jpg`, `03.jpg`, etc. This is a strict convention; don't use
  the original camera/export filename in `public/`.
- Source/unprocessed exports currently live at
  `C:\Users\idobl\Pictures\TirasPhotos\` (Ido's machine, flat folder,
  original export filenames preserved — e.g.
  `2025-09 - Greece - _C_C6273-export.jpg`). This path is specific to the
  current setup, not a project convention — if photos arrive from a
  different folder or person, just point the import step at wherever they
  actually are.

**Nothing scans automatically.** Every photo that should appear on the site
has to be explicitly listed once (in the import script's map, or by hand if
you're not using a script) — there's no "drop files in a folder and it just
works" behavior today.

## Processing pipeline (what happens to every real photo)

`scripts/import-photos.mjs` does the following to each source photo:

1. Corrects EXIF orientation (rotates upright), then strips all EXIF
   metadata (privacy + smaller file size)
2. Downscales if the long edge exceeds **2560px** — that's the largest
   responsive size the site ever requests (including the fullscreen
   lightbox), so anything bigger is wasted bytes
3. Re-encodes as JPEG, quality 92, mozjpeg encoder
4. Saves to `public/images/series/{category}/{NN}.jpg`

You don't need to pre-resize, pre-compress, or strip metadata yourself —
feed it full-resolution Lightroom/Capture One JPEG exports at whatever size
they come out at, and the script normalizes them. It prints the final
width/height/filesize for every photo it processes — **copy those exact
numbers into the JSON** (see below), don't estimate.

## The JSON schema — what every image entry needs

Each category's `content/series/{slug}.json` has an `images` array. Adding a
real photo means adding or updating one entry:

```json
{
  "src": "03.jpg",
  "layout": "single",
  "alt": "An 18th-century stone arch bridge over a still mountain stream, with reflection forming a perfect almond shape",
  "caption": "Kalogeriko bridge, Zagori",
  "width": 2560,
  "height": 1707
}
```

| Field | Required | Notes |
|---|---|---|
| `src` | yes | Filename only, no path. **String** for `full-bleed`/`single`. **Array of 2 strings** for `pair`. **Array of 3 strings** for `triptych`. |
| `layout` | yes | One of `"full-bleed"` \| `"single"` \| `"pair"` \| `"triptych"` — see Layout Guide below |
| `alt` | yes | String (or array matching `src`'s length for `pair`/`triptych`). See "Writing alt text" below — this isn't optional or decorative, it's read by screen readers and used as SEO/lightbox metadata. |
| `caption` | no | Short location/title shown in small caps under the image. Omit entirely for no caption, or use `"—"` if you want the caption line present but visually empty (existing placeholders do this). |
| `width` / `height` | yes | **Pixel dimensions of the actual processed file** — exactly what `import-photos.mjs` printed for that file. Required to prevent layout shift (CLS); if these don't match the real file, the image renders slightly stretched or squished. |

Category-level fields (`title`, `year`, `tags`, `description`, `cover`,
`order`) rarely need to change. One thing to keep in sync: `cover` is the
**filename** (not path) of whichever image should represent the category on
the home page and the `/work` index — it should normally match
`images[0].src`.

## Layout guide — which `layout` to use

- **`full-bleed`** — showcase width (capped to ~85% of viewport height so
  it's never taller than the screen). Use for the strongest single image in
  a category. 1–2 per category, max — overuse and it stops feeling special.
- **`single`** — centered, medium-sized, generous whitespace margins. The
  default. Most photos should be this.
- **`pair`** — two images side-by-side on desktop (stacked on mobile). Use
  only for genuinely related photos — two angles of the same subject, a
  visual rhyme, a clear before/after.
- **`triptych`** — three images in a row. Rare. Only when three photos
  truly belong together as a deliberate set.

**The first image in every category (`images[0]`) is special** regardless
of its `layout` value — it's rendered by `SeriesHero.tsx` as the page-opening
hero, sized to fit the viewport without scrolling, and is the element that
visually morphs in from the home page (the shared-element transition). Put
the strongest, most "establishing" photo there, and make sure its filename
matches `cover`.

## Step-by-step: adding a new batch of photos

1. **Sort** the new exports into the 5 categories by eye. Don't rename the
   original files yet.
2. **Decide the slot** for each: which category, and which `NN` (next
   available number in that category, or replacing a specific placeholder
   slot).
3. **Add to the import map.** Open `scripts/import-photos.mjs`, add one
   entry per photo to the `map` object:
   ```js
   "<exact source filename>.jpg": ["<category-slug>", "<NN>.jpg"],
   ```
4. **Run it:** `node scripts/import-photos.mjs` — review the printed
   width/height/filesize for each photo.
5. **Update the JSON.** For each new/changed photo, edit
   `content/series/{category}.json`: add or update the image entry with
   `alt`, `caption` (optional), and the exact `width`/`height` printed in
   step 4.
6. **Check `cover`** — if you changed what's at index 0, update the
   category's `cover` field to match.
7. **Build:** `npm run build` — confirms everything compiles and the new
   images are actually found.
8. **Look at it** — open `/work/{slug}` and check the result before calling
   it done.

## Writing alt text (and captions) in this site's voice

Two things matter equally: screen-reader usefulness, and matching the
quiet, observational tone already established (look at any existing entry
in `content/series/*.json` for the calibration). Avoid:

- Generic descriptions ("a nice photo of mountains")
- Marketing language ("stunning," "breathtaking," "magical")
- Camera-settings jargon in alt text (that's caption territory, not alt)

Aim for: specific, concrete description of what's actually in the frame —
enough that someone who can't see the image gets the same information a
sighted visitor gets at a glance. Example from the existing Landscape data:

> "An 18th-century stone arch bridge over a still mountain stream, with
> reflection forming a perfect almond shape"

Not:

> "Beautiful bridge photo"

Captions, when used, are short — a place name, a time, or nothing at all
("—"). They're not the place for the full story; that belongs in the
category's `description` field or the artist statement on
`/behind-the-lens`.
